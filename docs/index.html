The big picture — what does this thing do?
Imagine you're a security guard at a club entrance. Every person (pod/deployment) trying to get into the club (your Kubernetes cluster) has to pass through you first. You check their ID, their dress code, their guest list status. If they fail any check — rejected, no entry.

That's exactly what this webhook does. It sits between kubectl apply and the actual resource creation. Kubernetes asks your webhook: "should I allow this?" — and your code decides.

The 4 files that matter
The entire brain of this project is in 4 Go files. That's it. Everything else (Dockerfile, Helm, CI) is packaging.
config.go
Loads the YAML file that says which rules are on/off. Just a struct + a file reader.
validator.go
The brain. Every policy check lives here. This is where pods get approved or rejected.
handler.go
Receives the HTTP request from Kubernetes, unpacks it, sends it to the validator, packs the response back.
main.go
Starts the HTTPS server, loads TLS certs, handles graceful shutdown. The entry point.
3 of 9
config.go — the rulebook
This is the simplest file. It defines a Go struct where each field is a policy toggle, then reads a YAML file to fill it in.
type Policy struct { BlockLatestTag bool // reject :latest images RequireResourceLimits bool // must set CPU + memory RequireLabels []string // e.g. ["app","owner"] BlockedRegistries []string // e.g. ["untrusted.io/"] BlockPrivilegeEscalation bool // no privileged containers BlockHostNetwork bool // YOUR policy! MaxReplicaCount int // cap replicas ExemptNamespaces []string // skip kube-system etc }
The Load() function reads the YAML, the Default() function returns sane defaults if the file is missing. That's the entire file.

validator.go — the brain
This is where it gets real. When a pod comes in, ValidatePod() runs every check in sequence. Each check either passes silently or appends an error message. At the end — if there are zero messages, the pod is allowed.
The check sequence:
Exempt namespace?
→
Image tag check
→
Resource limits
Privilege check
→
Registry check
→
Label check
→
HostNetwork
The green one at the end is the policy you added.

Let's trace a real rejection
Someone runs: kubectl run bad-pod --image=nginx

What happens inside your webhook:
1. Image = "nginx" — no tag, no colon. isLatestOrUntagged() returns true
2. Resources.Limits = nil — no CPU, no memory set. Violation added
3. Labels = {} — missing "app" and "owner". Two violations added
4. Messages array has 4 items. len(messages) != 0 → Allowed = false
5. API server gets the denial → user sees the error in their terminal

handler.go — the translator
Kubernetes doesn't send you a Pod object directly. It wraps everything in an AdmissionReview envelope. The handler's job:
1. Read the raw HTTP body (JSON)
2. Unmarshal it into an AdmissionReview struct
3. Check req.Kind.Kind — is it a "Pod" or "Deployment"?
4. Unmarshal the inner req.Object.Raw into the right Go type
5. Call validator.ValidatePod() or ValidateDeployment()
6. Wrap the result back into an AdmissionResponse
7. Send it back as JSON
Think of it as a translator between "HTTP world" and "Kubernetes object world."
7 of 9
main.go — the server
This is the boring-but-important part. It does 4 things:
1. Loads config — reads policy.yaml, falls back to defaults
2. Loads TLS certs — K8s only talks to webhooks over HTTPS
3. Registers routes — /validate (the webhook) and /healthz (liveness probe)
4. Graceful shutdown — catches SIGTERM, drains connections cleanly
The TLS part is why the hack/gen-certs.sh script exists — you need certs to test locally.

The packaging — Dockerfile, Helm, CI
The Go code is the brain. Everything else gets it into a cluster:
Dockerfile
Multi-stage build. Compiles Go in alpine, copies binary to distroless. Runs as nonroot. Tiny, secure image.
Helm chart
Deployment, Service, ValidatingWebhookConfiguration, ConfigMap. One helm install and everything is wired.
GitHub Actions
Test → Lint → Build → Push to GHCR. Runs on every push to main. Full CI pipeline.

How to explain this in an interview
When someone asks "tell me about this project," here's your 30-second pitch:
"I built a Kubernetes validating admission webhook in Go that enforces deployment policies at the cluster level. It intercepts every pod and deployment creation, checks things like image tags, resource limits, privilege escalation, and required labels — and rejects anything that violates policy. Policies are configurable via YAML so teams can tune them without changing code. It's packaged with Helm, runs on distroless with TLS, and has a full CI pipeline in GitHub Actions."
Then they'll ask follow-ups. The steps you just walked through are the answers.
